# 视界之声（VisionVoice）技术研究报告（评审提交版）

## 文档信息

- 文档属性：项目评审材料（技术）
- 评审对象：视界之声（VisionVoice）后端与 Web 交互主链路
- 评审结论：有条件通过
- 版本：v1.0
- 日期：2026-03-16

---

## 1. 执行摘要

本报告基于项目计划书与现有代码实现，对 VisionVoice 的架构完整性、实现一致性、工程风险和可落地改造路径进行审查。结论如下：

1. 系统已形成可运行的端云协同能力，尤其在“流式文本转语音并行链路”和“会话打断隔离机制”方面具备较强工程价值。
2. 计划书核心能力基本落地，但存在“原型能力可用、生产治理不足”的阶段性特征。
3. 当前不建议直接进入大规模生产部署，建议以“有条件通过”方式进入整改闭环。
4. 准入前必须完成三项高优先级整改：密钥与凭据治理、密码安全存储、并发写一致性改造。

## 2. 研究范围与方法

### 2.1 研究目标

本报告重点回答以下评审问题：

1. 当前系统实际实现状态及关键技术路径。
2. 项目计划书与代码实现的一致性、偏差与影响面。
3. 架构、性能、安全、可维护性方面的主要风险分级。
4. 可执行的整改路线、验收标准与阶段目标。

### 2.2 研究范围

本次分析覆盖：

1. 项目计划书（Markdown 版本）。
2. back-end 目录下 Python 服务、静态脚本与配置模板。
3. 依赖文件与测试脚本。

不纳入本次结论的范围：

1. README 相关叙述性文档。
2. front-end 目录下快应用工程。

### 2.3 研究方法

1. 结构分析：识别入口、核心模块、依赖关系、关键运行路径。
2. 链路走读：按“认证 -> 会话 -> 语音 -> 视觉 -> 相册 -> 运维”逐段审查。
3. 对照分析：将计划书承诺项映射至可验证代码实现。
4. 风险评估：按高/中/低三级给出影响与整改优先级。

## 3. 系统总体技术架构（现状）

### 3.1 技术栈

1. 服务框架：Flask + Flask-SocketIO。
2. 身份认证：JWT（flask-jwt-extended）。
3. 大模型服务：智谱（文本、视觉多模态、情感模型）。
4. 语音服务：百度 ASR + 百度 TTS。
5. 视觉服务：
   - Python 侧 YOLO + EfficientNet（寻物/避障实验模块）。
   - 浏览器侧 TensorFlow.js YOLO + MobileNet（实时避障/寻物主流程）。
6. 配套能力：阿里云短信、文件系统存储、JSON 配置存储。

### 3.2 架构分层

1. 接入层：HTTP 路由（Flask）+ Socket 通道（音频片段下发、识别回传）。
2. 编排层：会话构建、模型路由、流式断句、异步 TTS 排队。
3. 能力层：ASR、TTS、LLM、多模态视觉与定位语义生成。
4. 存储层：用户信息（JSON）、图片/音频/文本文件、临时缓存目录。

### 3.3 关键业务链路

1. 普通对话：录音上传 -> ASR -> LLM 流式输出 -> 断句与异步 TTS -> 前端顺序播放。
2. 视频问答：图像上传 -> 多模态消息拼装 -> 视觉模型流式文本 -> 语音播报。
3. 有声相册：批量上传 -> 生成图像描述 -> 音频文本落盘 -> 前端点播与追问。
4. 生活助手：端侧视觉推理 -> 空间语义拼句 -> 统一音频播报通道。

## 4. 关键模块技术评审

### 4.1 服务入口与鉴权

1. 采用 before_request 进行入口过滤，对关键 API 执行 JWT 校验。
2. 白名单放行静态路径并拒绝未注册路由。
3. 登录/注册后返回 JWT，由前端在 Authorization 头携带。

评审意见：

1. 优点：接入链路清晰，移动端适配成本低。
2. 风险：鉴权路径硬编码维护成本高；密钥硬编码存在泄露隐患。

### 4.2 会话组织与模型调度

1. 普通文本使用 glm-4-plus-0111。
2. 视频/相册问答使用 glm-4v-plus-0111。
3. 心理陪护路由到 emohaa（含角色设定）。
4. 聊天历史按用户与 agent 维度存储，并做 Base64 编解码。

评审意见：

1. 优点：模型路由清晰，功能分层合理。
2. 风险：JSON 持久化在并发下存在写竞争与一致性风险。

### 4.3 流式文本到流式语音并行链路

1. 文本经 SSE 分块输出。
2. 分块进入断句器，识别句末停顿符。
3. 完整句子提交 AsyncTaskQueue 执行 TTS。
4. process_audio_stream 按句序出队，通过 Socket 分片推送。

有序机制：

1. 按 task_index 存储结果。
2. 仅按 current_index 拉取结果。
3. 前端再按 index 播放，形成前后端双重顺序保障。

评审意见：

1. 优点：首句时延明显改善，具备“边生成边播报”体验。
2. 风险：长文本场景下线程数量增长，缺少线程池上限控制。

### 4.4 打断与任务隔离

1. 前端按轮次递增 curr_task_id。
2. 上下行音频数据均携带 task_id。
3. 前端仅消费当前任务数据，旧任务自动失效。
4. 后端检测 task_id 变化后重置缓冲，避免串音。

评审意见：

1. 该机制有效解决“插话后旧回答继续播放”问题。
2. 该能力达到计划书“可打断可恢复”核心承诺。

### 4.5 语音识别与语音合成

1. ASR 使用百度极速接口，后端执行 wav 重采样。
2. TTS 使用 text2audio，支持语速参数与括号叙述过滤。
3. 前端在 agent.js/phone.js 中根据环境 dB 执行静音阈值校准。

评审意见：

1. 优点：噪声适配逻辑有效降低误触发。
2. 风险：外部接口重试与降级策略不完整；语速配置存在频繁 IO。

### 4.6 视觉能力（双栈并存）

1. Python 侧：YOLO + 测距/特征检索能力，偏实验和候选实现。
2. 浏览器侧：IndexedDB 缓存模型，端侧完成避障/寻物主流程。

评审意见：

1. 优点：端侧推理提升实时性并减轻后端算力压力。
2. 风险：双栈职责边界不清，Python 视觉主流程接线不充分。

### 4.7 有声相册与数据组织

1. 实现图片描述、音频生成、文本落盘与追问闭环。
2. 存储形态以文件系统 + JSON 为主。

评审意见：

1. 优点：功能闭环完整，用户感知价值高。
2. 风险：扩展后检索、审计、并发一致性能力不足。

## 5. 计划书一致性结论

### 5.1 高一致项

1. 流式文本与流式语音并行能力已实现。
2. 会话可打断机制已实现。
3. 多模态图像问答能力已实现。
4. 语音环境自适应能力已实现。
5. 端云协同模式已形成可运行方案。

### 5.2 部分一致项

1. 计划书强调后端视觉链路，当前主交互更偏端侧视觉推理。
2. 多用户并发具备基础支持，但未达到高并发一致性标准。

### 5.3 待闭环项

1. Python 视觉模块需明确“生产启用/实验保留”策略。
2. 安全与隐私治理尚未达到长期运营级标准。

---

## 6. 风险分级与处置建议

### 6.1 高风险（上线前必须整改）

1. 密钥管理：存在硬编码和固定配置路径风险。
2. 密码安全：用户密码明文存储于 JSON。
3. 数据一致性：并发写 user.json 缺少事务与锁机制。

### 6.2 中风险（近期整改）

1. 双视觉栈并存导致维护复杂度上升。
2. 外部服务调用缺少完善的超时/重试/降级策略。
3. 线程式 TTS 队列在极端场景可能出现资源抖动。

### 6.3 低风险（纳入常规治理）

1. 语速配置等局部代码重复。
2. 部分测试脚本与当前接口返回结构存在偏差。

## 7. 评审准入建议

### 7.1 准入结论

结论：有条件通过。

准入条件：

1. 完成 6.1 三项高风险整改并通过复核。
2. 提交最小可用的监控与审计方案（错误率、超时率、关键接口日志）。
3. 明确视觉主栈路线，给出淘汰或保留策略。

### 7.2 整改闭环表（提交评审建议附页）

## 8. 优化路线与里程碑

### 8.1 第一阶段（1-2周）

1. 修复高风险项：密钥治理、密码哈希、基础限流。
2. 补齐关键日志字段，清理重复代码。

### 8.2 第二阶段（3-6周）

1. 数据层迁移：JSON -> DB + Redis。
2. 任务执行迁移：裸线程 -> 受控异步/队列框架。
3. 落地统一错误码与健康检查。

### 8.3 第三阶段（6-10周）

1. 完成视觉栈收敛与模块治理。
2. 建立可观测体系（指标、追踪、告警）。
3. 组织真实用户 A/B 与盲测评估。

---

## 9. 量化评测指标（建议纳入评审看板）

1. 首句时延（P50/P95）。
2. 完整回答语音时延（含队列等待）。
3. 打断生效时间（用户打断到旧音频停止）。
4. ASR 有效率（有噪/无噪分桶）。
5. 寻物准确率、误报率、小目标召回率。
6. 避障距离误差（按距离区间统计）。
7. 系统稳定性（错误率、超时率、重试成功率）。

---

## 10. 综合结论

VisionVoice 在核心交互体验方面已具备较强可用性，尤其在流式语音并行与任务级打断机制上体现出明确工程亮点。当前短板主要集中在安全治理、数据一致性和可运维性，而非功能完整性本身。

在完成本报告列示的高风险整改后，项目可由“可演示原型”进入“可试点运营”阶段；在完成第二、三阶段治理后，可进一步具备规模化运营能力。
